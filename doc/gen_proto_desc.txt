usual protocol work start with this idea: 
WE ALREADY somehow know eacho other master public key (a.k.a ID)

what do we do if we don't know it? it differs...
this protocol have specific messages: intro_req, intro
it unreliable, coz we litteraly don't know with who we connecting
but it's useful for 1st time meeting
especially for server-nodes regarding their new clients (i hope)
and it's usage must be specified in node's config

first byte
00..E0 - encrypted msg, so, next goes nonce and MAC,
and then goes actual command it's 1st encrypted byte
E1..FF - plain unencrypted msg (even if actually it's semiencrypted)
actual cmd in 2nd byte.
but 2nd byte it's result of special operation between 1st byte and actual cmd
/* this operations probably will be deleted*/
